# 研究導向技術選型指南

## 前言

本文件是五階段研究式方法論的一個實用子集：

| Meta‑Guide    | Tech‑Stack‑Guide |
|---------------|------------------|
| 問題空間      | *(省略)*         |
| 抽象          | *(省略)*         |
| 建模          | 建模             |
| 原型          | 原型             |
| 產品          | 產品             |

**Meta‑Guide** 涵蓋了 *問題空間* 與 *抽象* 等概念性階段，這些主要是思考框架，而非程式實作。

相較之下，**Tech‑Stack‑Guide** 專注於工程相關的階段（*建模 → 原型 → 產品*），在這些階段中工具、語言與工作流程能直接應用。

額外的 *建模* 階段具有關鍵作用：它能在投入原型開發之前，先篩選掉不合適的問題，避免在概念上有缺陷的專案浪費開發資源。

技術選擇往往受到個人經驗與團隊偏好的影響。因此，本指南建議作為參考依據，並根據實際需求進行調整。

本文首先概述技術選型的整體方向，接著分析各種技術棧的優勢與限制。

## 技術選型

### 建模階段

* 先使用自由文字（free text）記錄靈感。
* 對於邏輯關係模型可使用 OCaml 建模。
* 對於數理計算模型可使用 Julia 建模。
* 若模型兼具邏輯關係及數理計算，則根據團隊偏好選擇 OCaml 或 Julia 建模。
* 若需要在 OCaml 中進行數理運算，可使用 `Lwt` 庫。
* 若需要在 Julia 中確認邏輯關係，可使用 `MLStyle` 庫。

### 原型階段

* **Workflow 專案**：採用 POSIX sh。
* **其他類型專案**：採用 Perl、Python、Ruby 或 Julia 之一。

### 成品階段

* 採用 OCaml、F# 或 Rust 之一。
* 若團隊不習慣 ML，也可以用 Java 或 C# 之一。

### 特定部署環境

* **瀏覽器與邊緣運算 (Edge Function) 專案**：採用 JavaScript。
* **內容網站專案**：採用 PHP。
* **企業專案**：採用 Java 或 C#。
* **行動應用專案**：採用 Dart。

> **備註**：只有在受限於特定部署目標時，才考慮使用部署棧。由於部署棧通常存在特定侷限或缺陷，因此不將其作為原型棧或成品棧。

### 舊專案 (Legacy Project)

* 以原語言維護舊專案。除非有明確理由，不進行重寫。
* 技術棧若停止更新或從系統中移除，需儘早遷移專案。
* 若部署環境淘汰、原技術棧無法繼續使用，需儘量遷移專案。
* 在資料結構、演算法正確的前提下，仍碰到效能瓶頸，可考慮遷移專案。

## 原型

* Perl、Python 與 Ruby 皆可作為原型棧，三者在核心語言能力與開發效率上不分伯仲。
* 初期建議以「單一腳本檔案」快速驗證想法，待規模擴大後再重構成專案結構。
* 由於皆屬於動態語言，大量錯誤直到運行期才會暴露，因此不適合作為成品棧。

## 成品

* 成品專案的核心重點在於長期穩定度。因此，應選擇開發迭代慢，但程式碼品質佳、維護成本低的技術棧。
* OCaml、F# 與 Rust 皆具備強大的編譯器，能在編譯階段進行靜態檢查，篩除絕大多數的常見錯誤。
* 建議透過社群標準的專案建置與管理工具生成樣板專案，隨後再將此樣板專案拷貝至主要的 Git 專案目錄中開始開發。

## 穩定開發環境

* 在 WSL 上使用主流 Linux 發行版的 LTS 版本，確保基礎環境穩定。
* 不同主機皆維持相同的發行版與 LTS 版本，減少心智切換成本。
* 透過 VSCode 遠端連線至 WSL，作為輕量級 IDE 使用。
* 使用 Vim 或 Emacs Keymap 延伸套件，加速程式碼編輯效率。
* 使用 `nano` 編輯環境設定檔，保持簡單直接。
* 除非有明確理由，不因更換技術棧而更換 IDE 或編輯器。

## 最小學習環境

* 學習新的技術棧時，入口檔案一律命名為 `program` 或 `Program`，再依技術棧決定副檔名。
* 練習語法特性時，始終使用同一個入口檔案，省去建立專案的額外負擔。
* 搭配樣板專案、編譯腳本與 Artifact 清除工具，加速學習與測試流程。
* 值得留存的範例程式碼保存於 GitHub 私人專案中，入口檔案本身不需保留。
* 只有在進一步學習如何建立應用程式或函式庫時，才需要深入使用技術棧的專案管理工具。

## 技術棧版本化

* 若技術棧有明確生命週期，選擇 LTS 版本而非最新版本。
* 若技術棧無明確生命週期，選擇穩定版本而非最新版本。
* 將語法與標準庫視為解決問題的工具，而非需要背誦的知識。
* 使用 LTS 或穩定版本的特性撰寫程式碼，以保持最大相容性。
* 不使用實驗性質特性撰寫程式碼，因為它們可能在未來被棄置。

## OCaml

* 內建高效的垃圾回收機制，開發者無需手動管理記憶體生命週期。
* 具備函數式程式語法特性與型別推導（Type Inference）。
* 語法設計極具表現力，能以簡短的程式碼實作複雜的邏輯。
* 以 OCaml 建模時，只要型態階層正確、編譯能通過就好，不用寫出完整程流程。
* 函式寫成 stub 即可，不需要實作完整功能。
* 善用 OCaml 的 Variant 和 Pattern Matching 來建立型態階層。

## Julia

* 透過 JIT 轉譯 LLVM，並自動調用 BLAS 等高效能數理運算庫，兼具原型及產品的特性。
* 採用漸進式型態標註，需要將專案轉成產品時，再補上型態標註即可。
* 透過結構及多重派發取代傳統物件系統。
* 將運算過程寫成函式有利於日後實作時有參考。
* 用小數量實字權充數據即可，不需使用原始數據。

## POSIX sh

* POSIX sh 是建構 workflow 專案速度最快的技術棧，但不適合用於其他類型的專案開發。
* 初期建議先以「單一腳本檔案」進行開發，待程式規模擴大、邏輯變複雜後，再重構成專案結構。
* 語言特性過於寬鬆，極易寫出「表面正常運作、實則暗藏 bug」的程式碼。開發時必須搭配 `ShellCheck` 等靜態分析工具進行程式碼檢查。
* Shell 腳本缺乏庫（library）的概念。若有跨專案複用的需求，需自行維護一組通用工具函式庫（utility functions）並在專案間共享。
* 遇到複雜邏輯或特殊需求時，可引入其他腳本語言作為 sidecar（輔助腳本）協助處理，但整體的 workflow 仍由 shell 腳本掌控。

```shell
# bin/cli
CWD=$(CDPATH='' cd -- "$(dirname -- "$0")" && pwd) || exit 1
. "${CWD}/../lib/utils.sh"
```

這段程式碼展示如何在 shell 腳本內引入庫。

## Perl

* 作為 Unix/Linux 的基本配置，幾乎所有作業系統皆有預裝。
* 其 Regex 語法與效能是業界的實質標準。
* 支援單行程式模式（One-liner），極適合快速的文字與檔案串流處理。

> **備註**：傳統 Perl 語法過於靈活，相對不易理解，但自 `5.36+` 版本引入現代化語法後已大幅改善，故將其列入推薦的原型棧中。

## Python

* 擁有最龐大的社群套件庫（Ecosystem），幾乎所有新興應用都會在第一時間提供 Python 支援。
* 語法結構簡潔，宛如可執行的虛擬碼（Pseudocode），極易閱讀與團隊維護。

## Ruby

* 語言設計以開發者體驗為中心，語法漂亮、具備表現力。
* 其 Regex 支援與效能幾乎與 Perl 居於同等地位。
* 對領域特定語言（DSL）友善，適合用來建構內部 DSL 工具。

## Rust

* 透過擁有權（Ownership）與借用檢查（Borrow Checker）機制，在不使用垃圾回收機制的情況下達成記憶體安全。
* 雖然偏向底層系統程式語言，但兼具了許多函數式與物件導向的高階語法特性。
* 具備零成本抽象（Zero-cost Abstraction）與極致效能，是新專案取代 C 或 C++ 的首選語言。

## Java / C#

* Java / C# 的本質是企業棧，其特色是常見問題都有相對應的庫，只需專注在開發核心邏輯。
* 企業棧採用 managed code，開發者不需要管理記憶體，滅少程式 crash 的次數。
* Java / C# 的編譯器保護程式碼品質的效果沒那麼顯著，但生態系完整，實作成品容易。
* 很少會出現只有其中一種技術棧才能解決的專案，選擇 Java / C# 多為團隊偏好。

## JavaScript

* 作為瀏覽器與邊緣運算的唯一原生語言，在這些領域中仍具備不可取代的地位。
* 由於許多潛在錯誤直到運行期才會暴露，因此不適合作為成品棧。
* 開發者需要適應 JavaScript 歷史留下的特異語法與行為特性，這在撰寫原型時容易帶來摩擦。
* 撰寫程式碼時必須時刻考量目標環境是瀏覽器還是 Node.js，並需要手動撰寫環境相容層（runtime glue），這對原型開發而言是無謂的負擔。
* 專案初始化配置相對繁瑣，且生態系中的工具鏈變動頻繁、缺乏長期一致的標準，會增加不必要的配置摩擦。

## PHP

* PHP 的本質是圍繞 Web 輸出發展的腳本語言，核心強項在於動態生成 HTML、XML、JSON 與圖檔等各式資料格式。
* 其核心功能高度聚焦於網頁模板與 Web 基礎建設，對於建構內容網站而言，仍相當實用且高效。
* 在原型開發階段，網頁模板與 Web 輸出屬於冗餘功能，因此不建議將 PHP 作為原型棧。
* 身為動態語言，大量錯誤僅在運行期才會暴露，因此不適合做為成品棧。

對於內容網站的建構，建議採取「工具化」的封裝策略：

> **實踐經驗**：可利用 PHP 開發靜態網站產生器（SSG）。開發完成後，後續所有內容網站專案皆直接用該 SSG，無需再撰寫新的 PHP 程式碼。這相當於將 PHP 封裝進單一工具中，既利用其特長，又避免重複開發。

## F#

* 雖然語法不完全相容，但 F# 能有效延續 OCaml 的技術生命與函數式開發思維，讓團隊技術棧得以平滑過渡。
* 企業級（Enterprise）專案的重點在於快速調用既有 API。而 .NET 作為主流的企業級技術棧，擁有極為龐大且成熟的商務功能套件庫。
* 由於 F# 程式碼的執行依賴 .NET 執行環境（Runtime），這在發布成品時會帶來額外的部署成本，因此不適合作為成品棧。

## Dart

* Dart 的設計初衷旨在取代 JavaScript，其語法結構可視為更完善、更現代化的 JavaScript 替代方案。
* Dart 的核心生態系高度集中於 Flutter 框架，因此極其適合作為行動裝置專案的特定部署環境。
* Dart 強制要求以專案架構進行程式碼編寫，這在快速驗證想法的原型階段會造成不必要的配置摩擦。
* 雖然其編譯器內建 null safety，但整體而言對程式碼品質與架構穩定度的提升幅度有限，不建議作為成品棧。

## C / C++

* 設計初衷旨在免除編寫組合語言的繁瑣，同時保有對底層系統與硬體的直接控制能力。
* 編譯器本身對程式碼品質的實質幫助較少，開發上依賴外部靜態分析工具來補強與改善程式碼品質。
* 開發者必須手動配置與釋放記憶體，這往往是系統安全漏洞與常見 bug 的主要來源。
* 以 C 語言而言，其內建的抽象機制相當匱乏，雖然給予開發者極大的自由度，但代價是須自行撰寫樣板程式碼。

## Go (Golang)

* 設計初衷是簡化 C / C++ 程式開發，並提供快速編譯與跨平台部署能力。
* 在雲端基礎建設（Kubernetes、Docker）與後端服務領域有廣泛應用。
* 語言特性刻意保持簡單，抽象能力有限。有時要撰寫樣板程式碼，不適合原型棧。
* 錯誤處理機制過於冗長，程式碼可讀性常受影響。
* 編譯器對程式品質的保障有限，仍需依賴外部工具，不適合成品棧。
